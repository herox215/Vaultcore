## What was fixed

### Bug 1 — Tree refresh after click-to-create (`54959b3`)
New `src/store/treeRefreshStore.ts` one-shot signal store. `Sidebar.svelte` subscribes on mount and calls `loadRoot()` on every new token. `EditorPane.handleWikiLinkClick` emits a refresh request after `createFile` resolves. Works around the watcher's `write_ignore` suppressing backend-initiated creates.

### Bug 2 — Autocomplete selected-row contrast (`e6a5da8`)
CM6's default theme (blue background + white text) was beating our class-only CSS override for background but losing for label color, producing low-contrast dark-on-blue. Forced brand accent (#6D28D9 purple) with `!important` plus white label + 75% white detail. High-contrast WCAG-friendly state that matches UI-SPEC brand.

### Bug 3 — Backlinks panel toggle button (`05e8ad1`)
Added a `PanelRight` (lucide-svelte) icon button to the editor topbar, right-aligned via flex spacer. Mirrors the existing left-sidebar collapse button pattern. `aria-pressed` reflects open state; active variant uses brand accent bg + color. Tooltip surfaces the keyboard shortcut. New users now discover the feature visually.

### Bug 4 — Rename-cascade race condition (`02d3483`) — **CRITICAL**
The most severe bug: rename-cascade silently wrote zero files. `update_links_after_rename` was reading `lg.incoming_for(old_path)` AFTER the disk rename, but the watcher had already dispatched `RemoveLinks(old_path)` to the LinkGraph consumer, emptying the incoming map. Users saw "successful" rename but link text stayed unchanged; the resolved-links client cache hid the problem until restart, when click-to-create resurrected the old filename.

Rewrote to iterate `FileIndex.all_relative_paths()` with rayon `par_iter()`, applying the `\[\[old_stem(\|alias)?\]\]` regex to each file content in parallel. Matches the RESEARCH.md Pattern 6 design originally intended. Fast-path `content.contains(old_stem)` check skips irrelevant files. All 18 LinkGraph unit tests still pass.

### Bug 5 — `[[` + closeBrackets conflict (`26ff492`)
`closeBrackets()` auto-inserted `]]` when user typed `[[`, producing `[[|]]` with cursor between. The autocomplete source bailed out on any `afterCursor.startsWith("]]")`, killing the popup in the most common trigger path. Users had to manually delete `]]` first.

Bail now only when cursor is inside a *completed* `[[Foo]]` (non-empty content AND `]]` follows). Apply string conditionally omits the closing `]]` when closeBrackets already inserted it, preventing `[[Filename]]]]` (four brackets).
