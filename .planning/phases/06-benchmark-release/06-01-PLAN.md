Extend `src-tauri/src/bin/gen_vault.rs` with the full generation pipeline. Structure:

1. **RNG setup:** `use rand_chacha::ChaCha20Rng; use rand::SeedableRng;` — seeded from `--seed`. All randomness routes through this one RNG so determinism holds.

2. **Markov model build (one-time per process):**
   - Split `WORDLIST` on `\n`, filter empty lines.
   - Build a bigram transition table: `HashMap<&str, Vec<&str>>` where the next-word bucket for word W is every word that follows W in a sliding window over the wordlist (the wordlist itself acts as the training text — this yields plausibly distributed tokens for Tantivy ranking per D-02).
   - Fallback: empty bucket → uniform-random next word. Prevents dead ends.

3. **Filename pool:** pre-generate `count` filenames upfront (`Note-{kebab-word}-{4-hex}`, with a few collisions planted — e.g. always-emit `Index.md` in 2+ folders). Hold them in a `Vec<PathBuf>` so body generation can sample wiki-link targets from a known set.

4. **Folder tree:** build hierarchy before emitting files.
   - Root has 3–8 top-level folders.
   - Each folder has depth 1–5 subfolders (Pareto — few deep, many shallow, per D-05).
   - Distribute filenames across folders via Pareto weight so middle levels get the bulk.

5. **Per-note body generation:**
   - Sample word count `n = max(10, round(Normal(μ, σ).sample()))` with `rand_distr::Normal`.
   - Sample link count from `link_density.0..=link_density.1`, tag count from `tag_density.0..=tag_density.1`.
   - Walk the Markov chain for `n - (link words + tag words)` tokens, joining with spaces, inserting paragraph breaks every 40–80 tokens.
   - Sprinkle wiki-links: pick `link_count` targets from the filename pool (Pareto so hub-notes exist — `rand_distr::Pareto` with shape α≈1.16 concentrates ~80% of links on ~20% of files, giving the 100+-backlink hubs from D-03). Render as `[[Target-Name]]`; occasionally `[[Target-Name|alias]]`.
   - Sprinkle tags: mix inline `#tag` + `#parent/child` and, for ~30% of notes, prepend a YAML frontmatter block:
     ```
     ---
     tags: [tag1, tag2/nested]
     created: 2026-04-13
     ---
     ```
   - On every 10k-th note (or 1 note total for small runs): emit a 10,000-line body (EDIT-08 sonder-kohort per D-04).

6. **Unresolved-link injection (D-06):** 5+ notes must carry `[[does-not-exist-XXXX]]` style targets the filename pool never includes.

7. **Edge-case pass after body generation:**
   - Convert ~5 notes from UTF-8 to Latin-1 bytes using `encoding_rs::mem::convert_utf8_to_latin1_lossy`-equivalent manual mapping (no new dep — iterate bytes, map non-ASCII chars through a tiny LUT, or just write a raw `[0x80, 0x81, 0xC4, …]` byte array that is valid non-UTF-8 — the indexer only cares that `std::str::from_utf8` fails).
   - Create 5+ symlinks via `std::os::unix::fs::symlink` on unix-family and `std::os::windows::fs::symlink_file` on windows (with `#[cfg(…)]`). On Windows, fall back to a plain copy + log a warning if symlinks not permitted (D-06 spec allows this).
   - Create `<out>/.obsidian/app.json` and `.obsidian/workspace.json` with minimal JSON content `{}`.

8. **Progress output:** on stderr every 1.000 notes: `"gen-vault: {i}/{count}"`. Redirectable, keeps CI logs thin.

9. **Determinism guard:** the entire run — including filename pool, folder assignment, body generation, edge-case file selection — must consume RNG values in a fixed deterministic order. No `HashMap` iteration order leaking into output. Use `BTreeMap` or sort keys before use.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI args → gen-vault process | Untrusted path input via `--out` |
| gen-vault → filesystem (user-controlled target dir) | Writes to arbitrary local path |
| Embedded wordlist → Markov model | Checked-in fixture, assumed trusted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01-01 | Tampering | `--out` path | mitigate | Canonicalize `--out`; refuse `/`, `/etc`, `$HOME` (top-level heuristics); require empty target unless `--force`; fs-writes use the canonicalized path only. |
| T-06-01-02 | Information disclosure | Symlink creation | mitigate | Symlink targets are restricted to paths inside the generated tree — never outside. Use relative symlinks only. |
| T-06-01-03 | Denial of service | `--count` arg | accept | User-triggered local tool; no network exposure. An oversized count is a self-DoS at most. Document default 100000 limit in `--help`. |
| T-06-01-04 | Spoofing | Wordlist integrity | accept | Wordlist is committed to VCS; Git history is the audit trail. |
| T-06-01-05 | Repudiation | Generation provenance | mitigate | `--version` emits deterministic sha256 used as CI cache-key, so a regenerated vault is traceable to the exact generator commit. |
| T-06-01-06 | Elevation of privilege | Symlink on Windows | accept | On Windows, symlink creation requires developer mode; we degrade to a copy with a stderr warning rather than request privileges. |

Severity: all low. No network, no privileged data. Generator only runs in dev/CI, never bundled into the Tauri app.
</threat_model>
